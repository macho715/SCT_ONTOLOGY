Spec v1.00 vs. actual repo — the key decision

The source spec describes a stack (Next.js + Vercel + Python FastAPI Worker) that contradicts what is already deployed (Cloudflare Workers + D1 + TypeScript MCP server). Pick the framing for the design we are about to build.

Spec v1.00 (proposed)

Stack

  • Vercel Next.js UI
  • Vercel API Orchestrator
  • Vercel Blob (private)
  • Python FastAPI Worker
  • SCT_ONTOLOGY = decision engine only
  • 7-sheet xlsx Audit Pack export
Actually deployed

Stack

  • Cloudflare Workers (already live)
  • D1 SQLite (already seeded)
  • TypeScript MCP server, 9 functions
  • Vitest test suite (30+ files)
  • OpenAI agent (Python) for ad-hoc queries
  • wrangler deploy, seed scripts, PII scanners

How should we frame the design?

A

Realign spec → existing Cloudflare stack

Treat the v1.00 spec as a wishlist. Rewrite it so Orchestrator = Workers routes, Worker = a Durable Object / Queue consumer, Blob = R2, xlsx = generated in-Worker. Smallest delta, fastest path to a working MVP.

B

Greenfield: build the Vercel + Python stack

Treat the existing CF/D1 server as a separate concern. Start a new Next.js project alongside it. Larger cost, but the v1.00 doc becomes a literal build plan.

C

Hybrid: keep CF/MCP, add a thin Next.js shell

UI = Next.js hosted on Vercel (upload + results + approval). MCP server stays on Cloudflare. Worker = a small Python service for xlsx + PDF. Splits risks but adds two deploys.

D

Step back: rethink what the platform actually is

The 7-sheet Audit Pack and AMBER/ZERO gates are the real product. The "Vercel vs Cloudflare" debate is downstream. Decide intent first, then pick stack.
