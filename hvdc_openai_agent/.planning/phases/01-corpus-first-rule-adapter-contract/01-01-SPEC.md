## Summary

### Problem

SCT_ONTOLOGY `ask_hvdc_ontology` currently answers from approved corpus evidence.
The Python `hvdc_openai_agent` contains useful deterministic sample shipment rules, but those rules are not yet exposed inside the existing answer flow.
Phase 1 must add a narrow adapter contract without letting sample shipment data become production corpus evidence.

### Goals

- G1: Add secondary shipment rule validation to the existing `ask_hvdc_ontology` data answer contract.
- G2: Preserve corpus-first evidence behavior for factual HVDC logistics answers.
- G3: Make rule-derived output visibly separate from `EvidenceSnippet` corpus evidence.
- G4: Keep `ask_hvdc_ontology` data-only and leave UI ownership with `render_hvdc_answer_card`.

### Non-Goals

- NG1: Do not create a new standalone MCP tool in Phase 1.
- NG2: Do not convert sample shipment data into corpus evidence.
- NG3: Do not mutate ERP, WMS, ATLP, Foundry, or any production system.
- NG4: Do not implement full Phase 2 validation signal merging.
- NG5: Do not change ChatGPT widget rendering ownership in Phase 1.

## User Scenarios & Testing

### User Story 1 - Matched shipment rule signal (Priority: P1)

An operations user asks an HVDC logistics question containing a known shipment key.
The answer keeps corpus evidence primary and returns a secondary `shipmentRule` result for the sample shipment match.

Why this priority: This is the core integration value for Phase 1.

Independent Test: Ask a question containing `BL-AUH-002` and assert that the answer includes `shipmentRule.found === true`, `shipmentRule.shipmentId === "SHP-0002"`, and unchanged corpus evidence semantics.

Acceptance Scenarios:

1. Given a question contains `BL-AUH-002`, When `ask_hvdc_ontology` builds the answer, Then `shipmentRule.found` is true and `shipmentRule.source` is `sample_shipment_rule_engine`.
2. Given a question contains `PKG-AGI-02`, When `ask_hvdc_ontology` builds the answer, Then the adapter can match the package key to sample shipment `SHP-0002`.
3. Given the adapter returns a sample shipment match, When the answer is assembled, Then the match is not added to `evidence[]`, `evidenceIds`, or `evidenceTrace.evidenceIds`.

### User Story 2 - Not-found shipment rule signal (Priority: P1)

An operations user asks a corpus-grounded question with a shipment key that does not exist in sample shipment data.
The answer returns a clear not-found `shipmentRule` state and does not let that state change the corpus verdict by itself.

Why this priority: Not-found behavior prevents false receipts, false claims, and unsupported operational closure.

Independent Test: Ask a question containing `NO-SUCH-BL` and assert that `shipmentRule.status === "INFO"` while the corpus verdict is not changed by this rule state alone.

Acceptance Scenarios:

1. Given no sample shipment matches a candidate key, When the adapter returns a result, Then `shipmentRule.found` is false and `shipmentRule.status` is `INFO`.
2. Given `shipmentRule.status` is `INFO`, When the final answer verdict is derived, Then the INFO not-found rule state does not change the corpus verdict by itself.

### User Story 3 - Unavailable rule adapter (Priority: P1)

The sample shipment data or rule adapter is unavailable.
The ontology answer still runs, and the rule result reports a WARN-style unavailable state.

Why this priority: The secondary rule signal must fail closed without breaking normal corpus answers.

Independent Test: Simulate unavailable sample rule data and assert that the answer contains `shipmentRule.status === "WARN"` and still returns the normal corpus answer path.

Acceptance Scenarios:

1. Given sample shipment data cannot be loaded or parsed, When `ask_hvdc_ontology` runs, Then `shipmentRule.status` is `WARN`.
2. Given the adapter is unavailable, When the question does not explicitly depend on secondary rule validation, Then normal corpus answer construction continues.

### User Story 4 - Data-only MCP contract remains stable (Priority: P1)

A ChatGPT App client calls `ask_hvdc_ontology`.
The returned data can include `shipmentRule`, but the ask tool still does not own UI metadata.

Why this priority: Current app contract separates data from rendering.

Independent Test: Run descriptor tests and assert that `ask_hvdc_ontology` has no `structuredContent.ui`, `_meta.ui.resourceUri`, or `openai/outputTemplate`.

Acceptance Scenarios:

1. Given `shipmentRule` is added to the answer schema, When descriptor parity tests run, Then `ask_hvdc_ontology` remains data-only.
2. Given render output is needed, When the app presents an answer card, Then `render_hvdc_answer_card` remains the presentation owner.

### Edge Cases

- EC-001: Multiple candidate identifiers are found; the adapter preserves ambiguity instead of silently choosing an unsafe target.
- EC-002: The question contains a package key not supported by current parent `resolveAnyKey()`; the adapter scans raw question text for package and shipment patterns.
- EC-003: Corpus evidence is missing or irrelevant; sample shipment data must not make an unsupported answer look supported.
- EC-004: Rule output is present but has no direct corpus support; it remains secondary sample validation only.

## Requirements

### Functional Requirements

- FR-001: The system MUST run shipment rule evaluation after corpus search has produced supported evidence and after `resolvedEntities` are available.
- FR-002: The system MUST run shipment rule evaluation before `validateGrounding()`.
- FR-003: The system MUST pass masked question text, `resolvedEntities`, and corpus-supported `evidence` into the adapter.
- FR-004: The system MUST add one optional `shipmentRule` object to `GroundedAnswer`.
- FR-005: `shipmentRule` MUST include at least `found`, `source`, `supportLevel`, `status`, `matchedKey`, `shipmentId`, `risks`, `humanGateRequired`, and `message`.
- FR-006: `shipmentRule.source` MUST identify the boundary as `sample_shipment_rule_engine`.
- FR-007: `shipmentRule.supportLevel` MUST identify the result as `SECONDARY_SAMPLE_VALIDATION`.
- FR-008: The system MUST NOT represent sample rule results as normal `EvidenceSnippet` corpus evidence.
- FR-009: When no sample shipment matches, the system MUST return `shipmentRule.found === false` and `shipmentRule.status === "INFO"`.
- FR-010: A not-found sample shipment result MUST NOT change the corpus answer verdict by itself.
- FR-011: When the adapter or sample data is unavailable, the system MUST return `shipmentRule.status === "WARN"`.
- FR-012: An unavailable adapter state MUST NOT break the normal ontology answer path unless the question explicitly depends on the secondary rule signal.
- FR-013: Identifier extraction MUST use both raw question text and existing `resolvedEntities`.
- FR-014: Identifier extraction MUST cover BL, BOE, DO, Invoice, Container, HVDC_CODE, Package, and shipment IDs.
- FR-015: The adapter SHOULD reuse parent `resolveAnyKey()` context when possible.
- FR-016: The adapter MUST scan raw question text for Python sample key patterns not covered by the parent router.
- FR-017: If multiple candidate identifiers are found, the adapter MUST preserve ambiguity rather than silently selecting one target.
- FR-018: The MCP output schema MUST accept optional `shipmentRule`.
- FR-019: `ask_hvdc_ontology` MUST remain data-only and MUST NOT attach UI metadata.
- FR-020: `render_hvdc_answer_card` MUST remain the owner of presentation metadata and answer card rendering.

### Non-Functional Requirements

- NFR-001 (Evidence Safety): Rule output must be visually and structurally separate from corpus evidence fields.
- NFR-002 (Determinism): The same question and same sample data must produce the same `shipmentRule` result.
- NFR-003 (Compatibility): Existing answers without `shipmentRule` must remain valid.
- NFR-004 (Failure Isolation): Adapter failure must be isolated to `shipmentRule.status === "WARN"` unless future phases explicitly define a blocking path.
- NFR-005 (Privacy): The adapter must use masked question text from the parent answer flow.
- NFR-006 (Testability): Found, not-found, unavailable, no-fake-evidence, and data-only descriptor behavior must each have an automated test.

## Assumptions & Dependencies

### Assumptions

- A-001: Phase 1 uses sample shipment rule data only.
- A-002: Sample shipment data is a secondary validation signal, not production transaction truth.
- A-003: Existing parent `ask_hvdc_ontology` corpus search, evidence support, and verdict derivation remain the primary answer path.
- A-004: Existing compatibility clients may receive answers without `shipmentRule`.
- A-005: The exact TypeScript enum or literal type names may vary if the public contract semantics remain stable.

### Dependencies

- DEP-001: `../server/src/answer.ts` for answer flow insertion.
- DEP-002: `../server/src/types.ts` for `GroundedAnswer` and answer contract types.
- DEP-003: `../server/src/index.ts` for MCP output schema and tool descriptors.
- DEP-004: `../server/src/router.ts` for existing `resolveAnyKey()` behavior.
- DEP-005: `src/hvdc_openai_agent/core.py` and `data/sample_shipments.json` for reference sample shipment behavior.
- DEP-006: `../tests/pipeline.test.ts` for behavior tests.

## Success Criteria

- SC-001: A question containing `BL-AUH-002` returns `shipmentRule.found === true`, `shipmentRule.shipmentId === "SHP-0002"`, and `shipmentRule.source === "sample_shipment_rule_engine"`.
- SC-002: A question containing `PKG-AGI-02` can match sample shipment `SHP-0002`.
- SC-003: A question containing `NO-SUCH-BL` returns `shipmentRule.found === false` and `shipmentRule.status === "INFO"`.
- SC-004: INFO not-found state does not change the corpus verdict by itself.
- SC-005: Simulated adapter or sample data unavailability returns `shipmentRule.status === "WARN"` without throwing to the caller.
- SC-006: Rule results do not add entries to `evidence[]`, `evidenceIds`, or `evidenceTrace.evidenceIds`.
- SC-007: `ask_hvdc_ontology` descriptor still has no `structuredContent.ui`, `_meta.ui.resourceUri`, or `openai/outputTemplate`.
- SC-008: Focused pipeline and descriptor tests pass.
- SC-009: Repository `npm run verify` passes from the parent SCT_ONTOLOGY root, or any inability to run it is recorded with the exact reason.

## Open Questions & Clarifications

### Open Questions

- OQ-001: Should an explicit user request for secondary shipment validation allow unavailable adapter state to become a future BLOCK condition? This is deferred to Phase 2 unless the owner changes scope.
- OQ-002: Should ambiguity be represented as `status: "INFO"` with candidates, or as a separate status value? Phase 1 may choose the exact literal only if ambiguity remains visible and testable.

### Clarifications Log

- 2026-05-11: User selected integration into existing `ask_hvdc_ontology`, not a separate tool.
- 2026-05-11: User selected corpus evidence first and shipment rule result as secondary validation.
- 2026-05-11: User selected adapter placement after corpus search and resolved entity context, before `validateGrounding()`.
- 2026-05-11: User selected single optional `shipmentRule` object shape.
- 2026-05-11: User selected not-found INFO and unavailable WARN handling.
- 2026-05-11: User selected raw question text plus `resolvedEntities` for identifier extraction.

## Risks & Mitigations

- R-001: Sample shipment data may be mistaken for production truth. Mitigation: require `source`, `supportLevel`, and no `EvidenceSnippet` conversion.
- R-002: Adapter failure may block normal answers. Mitigation: require WARN-style unavailable state and failure isolation.
- R-003: Rule-only claims may receive fake evidence IDs. Mitigation: test that `evidenceIds` and `evidenceTrace.evidenceIds` come only from corpus evidence.
- R-004: Optional answer field may break clients. Mitigation: make `shipmentRule` optional and update zod schema with descriptor tests.
- R-005: Multiple identifiers may cause unsafe target selection. Mitigation: preserve ambiguity in the adapter result.
