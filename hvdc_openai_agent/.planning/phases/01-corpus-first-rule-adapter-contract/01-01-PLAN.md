# Plan 01-01: Corpus-first shipment rule adapter contract

<objective>
Add a stable Phase 1 contract for secondary shipment rule validation inside the existing SCT_ONTOLOGY `ask_hvdc_ontology` answer flow.
The implementation must keep corpus evidence primary, add an optional `shipmentRule` object, and preserve the data-only tool contract.
</objective>

<must_haves>
<truths>
- D-01: Run the shipment rule adapter after corpus search has produced candidate evidence and before `validateGrounding()` runs.
- D-04: Add a single structured `shipmentRule` object to the answer data contract.
- D-09: Not-found sample shipment state is INFO-style and does not change corpus verdict by itself.
- D-11: Unavailable adapter state is WARN-style and does not break normal ontology answers.
- D-13: Use both raw question text and existing `resolvedEntities` to identify candidate shipment keys.
- CORP-01: Corpus evidence remains primary for factual HVDC logistics claims.
- CORP-02: Missing or irrelevant corpus evidence must not become supported only because sample shipment data exists.
- CORP-03: Rule-derived validation is labeled separately from corpus-supported evidence.
- RULE-01: Extract BL, BOE, DO, Invoice, Container, HVDC_CODE, Package, or shipment ID candidates.
- RULE-02: Resolve matching sample shipment data with deterministic any-key behavior.
- RULE-03: Return clear not-found state without changing verdict by itself.
- RULE-04: Fail closed when rule engine or sample data is unavailable.
- D-02: The adapter must receive the masked question, resolved entity context when available, and the corpus-supported state needed to avoid treating sample rule data as primary evidence.
- D-03: The adapter must not run before corpus retrieval because corpus-first behavior is a locked project decision.
- D-05: `shipmentRule` should include at least `found`, `source`, `supportLevel`, `status`, `matchedKey`, `shipmentId`, `risks`, and `humanGateRequired`.
- D-06: The `shipmentRule.source` value must make the boundary explicit, for example `sample_shipment_rule_engine`.
- D-07: The `shipmentRule.supportLevel` value must distinguish secondary sample validation from corpus-supported evidence.
- D-08: Do not represent sample rule results as normal `EvidenceSnippet` corpus evidence in Phase 1.
- D-10: A not-found sample shipment result must not change the corpus answer verdict by itself.
- D-11: If the adapter or sample data is unavailable, set `shipmentRule.status` to a WARN-style unavailable state.
- D-12: An unavailable rule adapter must not break the normal ontology answer path unless the question explicitly depends on the secondary rule signal.
- D-14: Reuse the parent pipeline's existing `resolveAnyKey()` context when possible.
- D-15: Also scan the question text for Python sample key patterns that may not exist in the current parent router.
- D-16: If multiple candidate identifiers are found, the adapter should preserve ambiguity rather than silently choosing an unsafe target.
</truths>
</must_haves>

<threat_model>

## Security And Safety Threat Model

| Threat | Severity | Mitigation |
|--------|----------|------------|
| Sample shipment data is mistaken for production truth | HIGH | `shipmentRule.source` must be `sample_shipment_rule_engine`; `supportLevel` must be `SECONDARY_SAMPLE_VALIDATION`; tests must prove `evidence[]` is not populated from rule results. |
| Adapter failure blocks normal ontology answers | MEDIUM | Unavailable state must be WARN-style and answer flow must continue unless future phases explicitly require hard blocking. |
| Rule-only claims get fake evidence IDs | HIGH | Do not add rule results to `EvidenceSnippet[]`; tests must assert evidence IDs come only from corpus evidence. |
| New answer field breaks render clients | MEDIUM | Make `shipmentRule` optional and update zod schemas before using it in answer objects. |
| Identifier ambiguity causes unsafe target selection | MEDIUM | Preserve multiple candidates in `shipmentRule` or return a clear ambiguous/not-found message rather than silently choosing. |

</threat_model>
