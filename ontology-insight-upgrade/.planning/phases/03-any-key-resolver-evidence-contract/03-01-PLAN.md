# Phase 3 Plan: Read-only Identifier Resolver with Evidence Contract

## Objective

Implement a read-only resolver that accepts supported operational identifiers and returns canonical context with confidence, evidence state, source fields, and privacy-safe evidence excerpts.

## Scope

In scope:

- Detect supported identifier schemes.
- Normalize identifier values.
- Resolve identifiers to canonical context from local fixture/index data.
- Return evidence references and source fields with every result.
- Return explicit states for ambiguous, missing, unsupported, and conflicting matches.
- Mask personal data, local paths, account IDs, and secrets in user-facing resolver output.
- Verify that resolver output keeps Phase 2 canonical semantics.

Out of scope:

- OAuth setup.
- Upload tools.
- Write tools.
- Remote MCP deployment changes.
- Cloudflare secret changes.
- Git repair.

## Task Breakdown

### Task 1: Write Resolver Contract

Create `docs/ANY-KEY-RESOLVER-CONTRACT.md`.

The contract must define:

- supported identifier schemes
- normalization rules
- resolver statuses
- confidence scale
- evidence-state shape
- source-field shape
- privacy-safe output shape
- canonical semantics boundary

Acceptance checks:

- Contract names every Phase 3 requirement.
- Contract states that resolver is read-only.
- Contract states that raw source evidence is not exposed by default.
- Contract references Phase 2 semantic adapter boundary.

### Task 2: Build Fixture Index

Create fixture data under `fixtures/resolver/`.

The fixture index must cover:

- exact HVDC code match
- invoice number match
- ambiguous container number match
- conflicting BL number evidence
- missing BOE number

Acceptance checks:

- Every fixture has expected output.
- Every expected output includes status, confidence, evidence refs, source fields, privacy metadata, and next action.
- No expected output uses legacy Flow Code as route lifecycle semantics.

### Task 3: Implement `hvdc_any_key_resolver.py`

Implement pure functions:

- `normalize_identifier_value(value: str) -> str`
- `detect_identifier_scheme(value: str, hints: dict | None = None) -> str`
- `mask_user_facing_value(value: str) -> str`
- `resolve_any_key(query: dict, index: dict | None = None) -> dict`

Implementation rules:

- Do not write files.
- Do not call network services.
- Do not mutate source fixtures.
- Use Phase 2 adapter helpers for no-evidence and uncertain-join responses where useful.
- Return deterministic JSON-serializable dictionaries.

Acceptance checks:

- Exact match returns `MATCHED`.
- Ambiguous match returns `AMBIGUOUS`.
- Conflict returns `CONFLICTING_EVIDENCE`.
- Missing key returns `NOT_FOUND`.
- Unsupported key returns `UNSUPPORTED_KEY`.
- User-facing evidence is masked.

### Task 4: Add Tests

Create `test_any_key_resolver.py`.

Tests must cover:

- supported scheme detection
- value normalization
- exact match resolution
- invoice match resolution
- ambiguous match behavior
- conflicting evidence behavior
- missing BOE behavior
- privacy masking
- canonical semantic guardrail compatibility

Acceptance checks:

- `python -m pytest test_any_key_resolver.py -v --tb=short` passes.
- Existing Phase 2 adapter tests still pass.

### Task 5: Add Guardrail Script

Create `scripts/verify-any-key-resolver.ps1`.

The script must run:

- Python syntax checks for the resolver.
- Resolver pytest tests.
- Semantic adapter pytest tests.
- Static marker checks for read-only resolver boundaries.

Acceptance checks:

- Script exits non-zero on failing tests.
- Script checks that no upload/write/OAuth mutation code exists in the resolver.
- Script prints a short PASS line only after all checks pass.

### Task 6: Update Operator Docs

Update `README.md` and `PRIVACY.md` only for user-facing resolver behavior.

Acceptance checks:

- README explains how to run the resolver verification command.
- Privacy doc explains what resolver output masks.
- Docs do not claim Cloudflare MCP write/upload/OAuth work is completed.

### Task 7: Write Completion Summary

Create `03-01-SUMMARY.md`.

The summary must include:

- changed files
- generated files
- test commands
- one-line result summary
- known limits
- next phase recommendation
