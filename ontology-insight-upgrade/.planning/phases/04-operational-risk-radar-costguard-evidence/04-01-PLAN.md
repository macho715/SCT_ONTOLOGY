# Phase 4 Plan: Read-only Risk Radar and CostGuard Evidence Pack

## Objective

Implement a read-only operational risk view and CostGuard evidence pack for a resolved shipment, package, invoice, or site context.

## Scope

In scope:

- Consume Phase 3 resolver output.
- Build structured risk cards for cost, invoice, tariff, VAT/duty, HS/OOG, customs, DEM/DET, milestone, document, and warehouse evidence.
- Build a CostGuard Evidence Pack for invoice-centered review.
- Include status, severity, confidence, evidence refs, missing inputs, owner cue, and one next action on every card.
- Preserve read-only behavior and evidence-review boundaries.
- Test exact evidence, missing evidence, and no-action boundaries.

Out of scope:

- MCP tool exposure.
- UI widget implementation.
- Approval, dispute, payment, email, escalation, graph write, or file upload.
- Cloudflare deployment changes.
- Git repair.

## Task Breakdown

### Task 1: Write Risk Radar Contract

Create `docs/OPERATIONAL-RISK-RADAR-CONTRACT.md`.

The contract must define:

- `OperationalRiskRadarResult`
- `RiskCard`
- `CostGuardEvidencePack`
- card severity and status vocabulary
- evidence ref requirements
- missing-input behavior
- owner cue behavior
- read-only and no-action boundaries
- Flow Code boundary

Acceptance checks:

- Contract covers all Phase 4 requirement IDs.
- Contract states that radar cards are evidence review aids, not operational actions.
- Contract states that CostGuard pack does not approve, dispute, or pay invoices.
- Contract states that missing evidence must remain visible.

### Task 2: Build Risk Evidence Fixtures

Create `fixtures/risk_radar/`.

The fixture index must cover:

- invoice-centered risk radar with CostGuard, VAT/duty, HS/OOG, customs, document, milestone, and warehouse evidence.
- site-centered risk radar with customs, milestone, document, warehouse, and DEM/DET evidence.
- CostGuard Evidence Pack with invoice line, BOE, CIPL, tariff, VAT, duty, and warehouse links.
- missing-evidence scenario that returns review stance without unsupported judgment.

Acceptance checks:

- Expected outputs include risk cards and evidence refs.
- Expected outputs include missing inputs when source evidence is absent.
- Expected outputs do not contain approval, dispute, payment, or mutation instructions.
- Expected outputs do not use Flow Code as route, customs, marine, cost, or KPI language.

### Task 3: Implement `hvdc_operational_risk_radar.py`

Implement pure functions:

- `build_operational_risk_radar(query: dict, resolver_index: dict | None = None, risk_index: dict | None = None) -> dict`
- `build_costguard_evidence_pack(query: dict, resolver_index: dict | None = None, risk_index: dict | None = None) -> dict`
- `rank_risk_cards(cards: list[dict]) -> list[dict]`
- `validate_risk_output(result: dict) -> list[str]`

Implementation rules:

- Do not write files.
- Do not call network services.
- Do not mutate graph data.
- Use Phase 3 `resolve_any_key`.
- Use Phase 2 `validate_public_output`.
- Return deterministic JSON-serializable dictionaries.

Acceptance checks:

- Invoice query returns `OperationalRiskRadarResult`.
- Site query returns `OperationalRiskRadarResult`.
- CostGuard query returns `CostGuardEvidencePack`.
- Missing-evidence query returns explicit `missingInputs`.
- Every risk card has one `nextAction`.

### Task 4: Add Tests

Create `test_operational_risk_radar.py`.

Tests must cover:

- invoice radar output
- site radar output
- CostGuard Evidence Pack output
- missing evidence behavior
- risk card required fields
- severity ranking
- no approval/dispute/payment claims
- read-only static behavior
- Phase 2 semantic guardrail compatibility
- Phase 3 resolver compatibility

Acceptance checks:

- `python -m pytest test_operational_risk_radar.py -v --tb=short` passes.
- Phase 3 resolver tests still pass.
- Phase 2 semantic adapter tests still pass.

### Task 5: Add Guardrail Script

Create `scripts/verify-operational-risk-radar.ps1`.

The script must run:

- Python syntax checks.
- Phase 4 radar tests.
- Phase 3 resolver tests.
- Phase 2 semantic adapter tests.
- Static checks for no write/upload/OAuth/action mutation behavior.
- Static checks that approval, dispute, and payment are not emitted as actions.

Acceptance checks:

- Script exits non-zero on failure.
- Script prints a short PASS line only after all checks pass.

### Task 6: Update Operator Docs

Update `README.md` with:

- what the risk radar does
- what the CostGuard Evidence Pack does
- the verification command
- explicit read-only boundary

Acceptance checks:

- README does not claim MCP public tool exposure.
- README does not claim approval, dispute, or payment automation.
- README points to the contract and verification command.

### Task 7: Write Completion Summary

Create `04-01-SUMMARY.md`.

The summary must include:

- changed files

- one-line result summary

- next phase recommendation
