<threat_model>
Trust boundaries:
- Local prototype query/rule shapes versus canonical HVDC MCP output.
- Evidence objects versus operational truth objects.
- Warehouse handling classification versus route, customs, marine, cost, and KPI language.
- Deterministic unit/contract tests versus live Fuseki checks.

STRIDE register:
- TM-01 Spoofing: Local `ex:Case` can look like the canonical shipment object. Mitigation: translate to canonical `ShipmentUnit` or `OperationalContext` before public output.
- TM-02 Tampering: Local fixtures can be edited to hide semantic drift. Mitigation: expected-output fixtures and file hashes in summary.
- TM-03 Repudiation: Adapter answers could omit evidence state. Mitigation: every canonical output must include evidence refs and one of PASS/WARN/BLOCK/NO_EVIDENCE/AMBIGUOUS/CONFLICT.
- TM-04 Information disclosure: Local paths, sample file names, or identifiers can leak through evidence payloads. Mitigation: keep public adapter output structured and redact local paths unless needed in internal test fixtures.
- TM-05 Denial of service: Live Fuseki dependency can block deterministic validation. Mitigation: Phase 2 tests use fixtures and mark live checks separately.
- TM-06 Elevation of privilege: Cost or warehouse evidence could be misread as approval/write authority. Mitigation: keep output read-only and block action, approval, dispute, payment, upload, and graph mutation fields.
</threat_model>
