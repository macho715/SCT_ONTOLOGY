# Phase 5 Plan: Read-only MCP Tool Surface and Release Verification

## Objective

Create a local, deterministic, read-only MCP-facing tool surface that wraps Phase 3 and Phase 4 capabilities and proves the V1 release boundary with contract tests.

## Planning Notes

No `CONTEXT.md` exists for this phase.
This plan uses the roadmap, requirements, Phase 5 research, and completed Phase 2 through Phase 4 contracts.

This plan does not claim live Cloudflare deployment, OAuth registration, ChatGPT connector registration, Claude registration, or Cursor registration.
Those require separate current-session verification.

## Scope

In scope:

- Define the V1 read-only MCP tool contract.
- Implement local MCP-style tool registry and invocation functions.
- Expose read-only wrappers for key resolution, risk radar, CostGuard evidence pack, evidence search, and validation.
- Return structured content with evidence references and operator-readable summaries.
- Validate malformed input, no evidence, secret redaction, read-only annotations, and no-action boundaries.
- Keep raw SPARQL, local Flask, local Fuseki, ngrok, upload, write, OAuth, email, payment, approval, dispute, and escalation out of the V1 surface.

Out of scope:

- Cloudflare Worker or Agent deployment.
- OAuth client registration.
- R2 upload tools.
- Graph write or file write tools.
- Public endpoint creation.
- UI widget work.
- Git repair.

## Task Breakdown

### Task 1: Write Read-only MCP Surface Contract

Create `docs/READ-ONLY-MCP-SURFACE-CONTRACT.md`.

The contract must define:

- V1 tool inventory
- tool input schemas
- tool output envelope
- structured content requirements
- evidence reference requirements
- operator-readable summary requirements
- read-only annotations
- privacy and secret-redaction expectations
- no raw SPARQL, local Flask, local Fuseki, ngrok, upload, write, OAuth, and action boundaries
- future upload/write/action separation rules

Acceptance checks:

- Contract covers `MCP-01`, `MCP-02`, `MCP-03`, `MCP-04`, `MCP-05`, and `TEST-04`.
- Contract states that V1 tool calls cannot mutate graph, shipment, invoice, warehouse, email, approval, or file state.
- Contract states that future upload/write/action tools require scopes, human gates, audit records, rollback design, and separate tests.
- Contract states that local contract proof is not the same as Cloudflare deployment proof.

### Task 2: Build MCP Surface Fixtures

Create `fixtures/mcp_surface/`.

The fixtures must cover:

- successful key resolution
- successful risk radar
- successful CostGuard evidence pack
- evidence search
- validation tool output
- malformed input
- unsupported or no-evidence key
- secret and local-path redaction

Acceptance checks:

- Every query fixture has an expected output fixture.
- Expected outputs include `structuredContent`, `content`, `evidenceRefs`, `validation`, `annotations`, `privacy`, and `actionBoundary`.
- Expected outputs include `readOnly: true` and `mutationAllowed: false`.
- Expected outputs do not expose raw SPARQL, local file paths, secrets, bearer tokens, ngrok URLs, Flask URLs, or Fuseki update URLs.

### Task 3: Implement `hvdc_readonly_mcp_surface.py`

Implement pure functions:

- `list_mcp_tools() -> list[dict]`
- `call_mcp_tool(name: str, arguments: dict | None = None) -> dict`
- `resolve_operational_key(arguments: dict) -> dict`
- `get_operational_risk_radar(arguments: dict) -> dict`
- `get_costguard_evidence_pack(arguments: dict) -> dict`
- `search_evidence_refs(arguments: dict) -> dict`
- `validate_mcp_output(arguments: dict) -> dict`
- `validate_readonly_mcp_surface_output(result: dict) -> list[str]`

Implementation rules:

- Do not write files.
- Do not call network services.
- Do not start local servers.
- Do not import Flask app, Fuseki deployment, ngrok setup, dashboard launchers, or GPTs Actions helpers.
- Use `resolve_any_key`.
- Use `build_operational_risk_radar`.
- Use `build_costguard_evidence_pack`.
- Use Phase 2 public-output validation where useful.
- Return deterministic JSON-serializable dictionaries.

Acceptance checks:

- `resolve_operational_key` returns matched, ambiguous, conflict, unsupported, or no-evidence states without forced joins.
- `get_operational_risk_radar` returns structured risk cards and evidence refs.
- `get_costguard_evidence_pack` returns evidence review only.
- `search_evidence_refs` returns evidence metadata only, not raw source rows or local files.
- `validate_mcp_output` returns validation findings without mutating anything.
- Unknown tool names return a structured error, not an exception traceback.

### Task 4: Add Contract Tests

Create `test_readonly_mcp_surface.py`.

Tests must cover:

- tool inventory includes all V1 read-only tools
- each expected fixture output matches exactly
- malformed input returns structured error
- no-evidence and unsupported-key states remain visible
- every output contains read-only annotations
- every output blocks mutation/action language
- secret, token, account-id, ngrok URL, localhost URL, and local path redaction
- raw SPARQL and local Flask/Fuseki output exposure are blocked
- Phase 3 resolver regression still passes
- Phase 4 risk radar regression still passes

Acceptance checks:

- `python -m pytest test_readonly_mcp_surface.py -v --tb=short` passes.
- Phase 4 radar tests still pass.
- Phase 3 resolver tests still pass.
- Phase 2 semantic adapter tests still pass.

### Task 5: Add Release Verification Script

Create `scripts/verify-readonly-mcp-surface.ps1`.

The script must run:

- Python syntax checks.
- Phase 5 MCP surface tests.
- Phase 4 risk radar tests.
- Phase 3 resolver tests.
- Phase 2 semantic adapter tests.
- Static checks for no Flask/Fuseki/ngrok/OpenAPI/GPTs Actions exposure from the MCP surface.
- Static checks for no write/upload/OAuth/payment/approval/dispute/email/escalation behavior.
- Static checks for read-only annotation markers.
- Static checks for secret redaction fixture coverage.

Acceptance checks:

- Script exits non-zero on failure.
- Script prints a short PASS line only after all required checks pass.
- Script reports live-service checks separately if any are run.

### Task 6: Update Operator Docs

Update `README.md` with:

- clear statement that this is a local contract surface unless separately deployed

Acceptance checks:

- README does not tell users to use local Flask, raw SPARQL, ngrok, or Fuseki as the V1 production surface.
