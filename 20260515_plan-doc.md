## A. Executive Summary

### A1. 목표
운영 중인 HVDC Ontology MCP 서버(Cloudflare Workers, 15 tools, 209 tests)에 **3가지 운영 강화 계층**을 추가한다:
1. **Axiom OTLP 연결** — 이미 구현된 `withSpan` OTel 코드를 실제 관찰 가능성 백엔드에 연결
2. **Hybrid 3-Tier Rate Limiting** — ChatGPT/Claude 정상 사용자 보호 + 악의적 호출 차단
3. **Tiered KV Cache** — D1 read 60-80% 절감, 데이터 변동성 기반 TTL 적용

### A2. 핵심 결정 (Confirmed)

| 결정 | 선택 | 이유 |

| OTLP backend | **Axiom** | 무료 500GB/월, 동일 otel-cf-workers 설정, $0-30/월 |

## B. Context & Requirements

### B1. 문제 정의

**문제 1 — OTel Dead End:**  
`OTEL_ENABLED=true`이고 `withSpan("ask_hvdc_ontology", ...)` 코드가 완전히 구현되어 있지만, `OTLP_ENDPOINT` wrangler secret이 설정되지 않아 span이 `/dev/null`로 사라지고 있다. 운영 중 BLOCK verdict 급증 같은 이상 징후를 감지할 수 없다.

**문제 2 — 무방어 엔드포인트:**  
`/mcp`에 rate limiting이 없다. `ask_hvdc_ontology`는 내부적으로 LLM 호출을 포함하며, 반복 호출 시 D1 quota와 Workers CPU 비용이 선형으로 증가한다. ChatGPT plugin 연동 시 비정상 retry loop 한 건으로 일일 quota를 소진할 수 있다.

**문제 3 — D1 전량 조회:**  
`resolve_any_key`, `check_mosb_gate` 등 read 전용 tool들이 매 호출마다 D1을 직접 조회한다. 제어탑 마스터 데이터(장비 코드, 마일스톤 등)는 시간당 수 회 이상 변경되지 않으므로 KV 캐시로 D1 부하를 크게 줄일 수 있다.

## C. MCP Tool Interface Design

> MCP 서버는 UI가 없다. 이 섹션은 tool schema, 응답 형식, 에러 메시지 UX를 다룬다.

### C1. Tool 인터페이스 현황

| Tool | 현재 | 변경 |
|---|---|---|
| `ask_hvdc_ontology` | 직접 D1 + OTel (span export 없음) | OTel span → Axiom 전송 |
| `resolve_any_key` | D1 직접 조회 | KV HIT → D1 MISS 순서 |
| `check_mosb_gate` | D1 직접 조회 | KV 캐시 적용 |
| 전체 `/mcp` | rate limit 없음 | 3-tier 미들웨어 |

### C2. 에러 응답 표준화

Rate limit 초과 시 MCP 프로토콜 준수 JSON 응답:

```json
{
  "error": "rate_limited",
  "message": "Too many requests. Please retry after 60 seconds.",
  "retryAfter": 60
}
```

HTTP 헤더: `Retry-After: 60`, `X-RateLimit-Limit: 200`, `X-RateLimit-Remaining: 0`

### C3. 캐시 상태 투명성

KV 캐시 히트 시 span attribute 추가 (운영 관찰용, tool 응답 변경 없음):

```typescript
span.setAttribute("hvdc.cache_hit", true);
span.setAttribute("hvdc.cache_key", cacheKey);
span.setAttribute("hvdc.cache_ttl_remaining", ttlRemaining);
```

### C4. Rate Limit 응답 흐름

```
Client → /mcp
  ├── Authorization: Bearer <token>
  │     ↓
  │   RATE_LIMITER.limit({ key: `token:${token}` })
  │     ├── success=true → proceed to MCP handler
  │     └── success=false → 429 { error: "rate_limited" }
  │
  └── (no token / anonymous)
        ↓
      RATE_LIMITER_IP.limit({ key: `ip:${cf-connecting-ip}` })
        ├── success=true → proceed
        └── success=false → 429

  Additionally for ask_hvdc_ontology tool:
    RATE_LIMITER_TOOL.limit({ key: `tool:ask:${token}` })
      10 calls/hour per token
```

## D. System Architecture

### D1. 컴포넌트 다이어그램

```
┌────────────────────────────────────────────────────────┐
│              Cloudflare Workers Runtime                │
│                                                        │
│  Request → [Rate Limit Middleware] ─────────────────┐ │
│              │                                       │ │
│              ▼ (pass)                                │ │
│         [MCP Handler]                                │ │
│              │                                       │ │
│    ┌─────────┴──────────┐                            │ │
│    │                    │                            │ │
│    ▼                    ▼                            │ │
│  [Read Tools]      [Write Tools]                     │ │
│  resolve_any_key   write_file_dry_run                │ │
│  check_mosb_gate   write_file_commit                 │ │
│    │                    │                            │ │
│    ▼                    ▼                            │ │
│  [KV Cache Layer]  [R2/D1 Direct]                   │ │
│    ├─ HIT: return                                    │ │
│    └─ MISS: D1 → KV.put(write-through)              │ │
│                                                      │ │
│  [OTel: withSpan]                                   │ │
│    └─ OTLP HTTP export → Axiom                      │ │
│                                          429 ────────┘ │
└────────────────────────────────────────────────────────┘

Bindings:
  HVDC_CACHE     KVNamespace     (read-heavy cache)
  MCP_AUDIT_DB   D1Database      (durable store)
  HVDC_FILES     R2Bucket        (file store)
  RATE_LIMITER       RateLimit   (token: 200/60s)
  RATE_LIMITER_IP    RateLimit   (IP: 2000/60s)
  RATE_LIMITER_TOOL  RateLimit   (tool: 10/3600s)
```

### D2. 데이터 흐름

**Read path (cached):**
```
Tool call
  → cachedD1Query(kv, d1, key, sql, params, CACHE_TTL.SHIPMENT_STATUS)
    → kv.get(key, "json")   // ~5ms global
      ├── HIT  → return JSON
      └── MISS → d1.prepare(sql).first<T>()  // ~20ms
                  → kv.put(key, json, { expirationTtl })  // write-through
                  → return result
```

**Write path (cache invalidation):**
```
write_file_commit / update operation
  → d1 write
  → kv.put(key, newData, { expirationTtl })  // write-through (NOT delete)
```
> `kv.delete()` 사용 금지 — 60초 global consistency window 내 stale 응답 위험

### D3. Rate Limit 아키텍처

```
Tier 1: per-Bearer-token    200 req/min  primary guard (ChatGPT plugin)
Tier 2: per-IP (fallback)  2000 req/min  anonymous/unauthenticated abuse
Tier 3: per-token+tool       10 req/hr   ask_hvdc_ontology LLM gating
```

- Tier 2 한도를 높게 설정한 이유: NAT 뒤 다수 사용자가 동일 IP를 공유하는 false positive 방지
- Tier 3는 Tier 1 통과 후 tool 실행 직전에 체크 (토큰 없으면 skip)

## J. Error Handling & Recovery

### J1. 에러 분류

| 에러 | 처리 | 사용자 응답 |
|---|---|---|
| `AXIOM_TOKEN` 미설정 | 경고 로그, no-op export | 정상 응답 (span 손실) |
| Rate limit 초과 | 즉시 429 반환 | `{ error: "rate_limited", retryAfter: 60 }` |
| KV `get` 실패 | fallback to D1 | 정상 응답 (캐시 없이) |
| D1 `prepare` 실패 | throw → MCP error | `{ error: "internal_error" }` |
| RATE_LIMITER binding 없음 | 경고 로그, bypass | 정상 응답 (rate limit 없음) |

### J2. 에러 처리 코드 패턴

```typescript
// KV fallback pattern (절대 KV 실패가 D1을 막으면 안 됨)
async function cachedD1Query<T>(
  kv: KVNamespace, d1: D1Database,
  cacheKey: string, sql: string, params: unknown[], ttl: number
): Promise<T | null> {
  try {
    const hit = await kv.get<T>(cacheKey, "json");
    if (hit !== null) return hit;
  } catch (e) {
    console.warn("[cache] KV get failed, falling back to D1", e);
  }
  const row = await d1.prepare(sql).bind(...params).first<T>();
  if (row) {
    try {
      await kv.put(cacheKey, JSON.stringify(row), { expirationTtl: ttl });
    } catch (e) {
      console.warn("[cache] KV put failed", e);
    }
  }
  return row ?? null;
}
```

### J3. Retry 정책

| 작업 | Retry | 이유 |
|---|---|---|
| KV get | No retry | 실패 시 D1 fallback |
| D1 query | No retry (CF 내부 처리) | Workers runtime이 처리 |
| OTLP export | Library 처리 (async) | otel-cf-workers 자동 재시도 |
| Rate limit check | No retry | 결정적 응답 |

### J4. Idempotency

- `cachedD1Query`: 읽기 전용, 자연히 idempotent
- `kv.put` write-through: 동일 키에 덮어쓰기 = safe
- Rate limit 체크: stateful (CF 글로벌), 멱등 아님 — 클라이언트가 retry 책임

### J5. Rollback Triggers

| 트리거 | 롤백 방법 | 소요 시간 |
